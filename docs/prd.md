# Team Lite Chat — 製品要求仕様書（PRD）

**ドキュメント種別:** プロダクト要求仕様（MVP 準拠）  
**プロダクト名:** Team Lite Chat（Slack 簡易版）  
**関連:** [architecture.md](architecture.md) · [IMPLEMENTATION_GUIDE.md](IMPLEMENTATION_GUIDE.md)

---

## 1. 概要

### 1.1 ビジョン（製品の一文）

ワークスペース内で**公開チャンネル**によるチーム会話を行える、Slack ライトな Web アプリ。将来的にはダイレクトメッセージ等へ拡張可能とするが、**MVP ではチャンネル中心**とする。

### 1.2 目的

- チームの非同期・準同期コミュニケーションを一箇所に集約する。
- メールやチャットツールの乱立を減らし、オンボーディングコストの低い MVP を提供する。

### 1.3 対象ユーザー・規模

- **規模:** 数十人規模のチームを主想定（数百人まで設計上は許容し、SLO は後続で定義）。
- **利用形態:** ブラウザ（レスポンシブ Web）。ネイティブアプリは対象外（MVP）。

### 1.4 ホスティング上の前提（確定事項）

- **実行基盤:** Next.js（App Router）を **Vercel**（サーバーレス）上でホストする。
- **データベース:** マネージド **PostgreSQL**（Neon / Supabase / Vercel Postgres 等）。`DATABASE_URL` で接続。
- **リアルタイム:** 長寿命 **WebSocket は採用しない**。**約 2 秒間隔のポーリング**と `?since=` による差分取得で準リアルタイムを満たす。将来、Ably / Pusher / Supabase Realtime 等への置換を可能とする。

## 2. ゴールと成功指標

### 2.1 プロダクトゴール（MVP）

1 ワークスペースに複数ユーザーが参加し、**同一チャンネルでメッセージの送受信が実用的な遅延**（目安: 体感 2 秒以内の更新。環境依存のため数値 SLO は後続）で行えること。

### 2.2 成功指標（定性的）

| ID | 指標 |
|----|------|
| S1 | 主要フロー（登録 → ワークスペース参加または作成 → チャンネル投稿 → 他ユーザーが近い将来に閲覧）が、別紙マニュアルなしで完了できる。 |
| S2 | ページリロード後も過去メッセージが一貫して表示される（DB 永続化）。 |
| S3 | 他ワークスペースのデータに、未認証・非メンバーが API 経由でアクセスできない。 |

## 3. スコープ

### 3.1 MVP に含める（In Scope）

| 領域 | 要求 | 実装メモ |
|------|------|----------|
| アカウント | メール + パスワードの登録、ログイン、ログアウト | bcrypt ハッシュ、Auth.js Credentials + JWT |
| ワークスペース | 作成、名前・スラッグ表示、**複数 WS への所属** | URL は `/w/[slug]`。作成時に `#general` 自動生成 |
| メンバー | **メンバー一覧の閲覧**（同一 WS 内） | **現行リリースでは未実装**。データは `Membership` で保持済み。次イテレーションでサイドバー等に表示。 |
| 招待 | **招待リンク**（有効期限付き）。メール招待は必須としない | `/join/[token]`、トークン発行 API |
| チャンネル | 公開チャンネルの作成・一覧。**WS 内で全メンバーが利用可能なチャンネル** | 命名規則はサーバー検証 |
| メッセージ | プレーンテキスト投稿、時系列表示、履歴取得（最新 N 件 + 差分ポーリング） | 最大 8000 文字 |
| 準リアルタイム | 閲覧中ユーザーの新着反映 | ポーリング + メッセージ ID によるクライアント側去重 |
| UI | 左サイドバー（チャンネル等）+ メインタイムラインの 2 ペイン構成 | — |

### 3.2 MVP の明示的対象外（Out of Scope）

- スレッド、リマインダー、ワークフロー、Bot、サードパーティ連携
- 音声・ビデオ通話、画面共有
- **ダイレクトメッセージ（DM）**（ビジョンには含めるが MVP ではチャンネルのみ）
- 高度な全文検索（Elasticsearch 等）
- ネイティブモバイルアプリ（レスポンシブ Web のみ）
- eDiscovery、データ保持ポリシー UI、コンプライアンス特化機能
- 絵文字リアクション、ファイルアップロード（「MVP+」として別イテレーション）

## 4. ユーザーストーリー

1. **新規ユーザー**として、ワークスペースを作成し、最初のチャンネル（`#general`）で挨拶を投稿したい。
2. **招待されたユーザー**として、リンクから参加し、チャンネル一覧から会話に加わりたい。
3. **メンバー**として、投稿した内容が他のオンラインユーザーに**短い遅延**で見えるようにしたい。
4. **管理者**として、不適切利用を抑止したい（**アカウント無効化は後続**でも可）。

## 5. 機能要件（詳細）

### 5.1 認証・セッション

| ID | 要件 |
|----|------|
| A1 | パスワードはサーバー側で業界標準的な方式でハッシュ保存する（bcrypt 等）。 |
| A2 | セッション方式は **JWT** とする。有効期限は実装で定義（例: 30 日）。 |
| A3 | 本番では HTTPS を前提とし、ホスト検証は `trustHost` および `AUTH_URL` で整合させる。 |

### 5.2 ワークスペース

| ID | 要件 |
|----|------|
| W1 | 1 ユーザーは**複数ワークスペース**に所属できる。 |
| W2 | ワークスペース識別子は **URL パス上のスラッグ**（例: `/w/acme-corp`）とする。 |
| W3 | 作成者はオーナー相当のロールを持つ（`role: owner` 等）。 |

### 5.3 チャンネル

| ID | 要件 |
|----|------|
| C1 | チャンネル名は **ワークスペース内で一意**。 |
| C2 | 命名規則: 英小文字・数字・ハイフン。正規表現例: `^[a-z0-9][a-z0-9-]{0,78}$`（実装と一致）。 |
| C3 | MVP のチャンネル作成権限は **WS メンバー全員**とする（管理者限定は後続）。 |

### 5.4 メッセージ

| ID | 要件 |
|----|------|
| M1 | 本文はプレーンテキスト。最大文字数は **8000**（実装と一致。PRD 上 4000〜16000 のレンジ内）。 |
| M2 | **メンション `@user` は MVP 対象外**（後続）。 |
| M3 | 編集・削除ポリシー: MVP では**未実装（削除のみ／編集のみ等はバックログで決定）**。 |

### 5.5 準リアルタイム・オフライン

| ID | 要件 |
|----|------|
| R1 | 接続断・タブ非表示後の再表示でも、再取得により一貫した履歴が得られること。 |
| R2 | クライアントは **メッセージ ID** ベースで重複表示を防ぐこと。 |

### 5.6 通知

| ID | 要件 |
|----|------|
| N1 | ブラウザプッシュ通知は **任意（MVP では未実装可）**。 |

## 6. 非機能要件

| 領域 | 要件 |
|------|------|
| セキュリティ | HTTPS、入力バリデーション、**他 WS のチャンネル・メッセージへの越境アクセス禁止**（401/403）。レート制限は理想としてログイン・投稿に適用（MVP では未実装可、バックログ明記）。 |
| 可用性・拡張 | メッセージは DB に永続化。リアルタイムは別プロセスに依存しない（ポーリング）。 |
| プライバシー | 保存する個人情報: メール、表示名、パスワードハッシュ、メッセージ本文、メタデータ。**保持期間・GDPR 対応は本 PRD では対象外と明記**（必要なら別紙）。 |
| アクセシビリティ | キーボード操作・コントラストは可能な範囲で準拠。詳細基準は後続。 |

## 7. データモデル（概念）

実体は Prisma スキーマが正とする。概念上は以下の関係。

- **User** — `email`（一意）、`name`、`passwordHash`
- **Workspace** — `name`、`slug`（一意）
- **Membership** — `userId` + `workspaceId`（一意）、`role`
- **Channel** — `workspaceId` + `name`（一意）
- **Message** — `channelId`、`authorId`、`body`、`createdAt`
- **Invite** — `workspaceId`、`token`（一意）、`expiresAt`（任意）

## 8. API・イベント方針

### 8.1 REST（HTTP）

認証、ワークスペース、チャンネル、メッセージ、招待の **Route Handlers** により提供。認可ルールはリソースごとに統一する。

### 8.2 イベント（リアルタイムの代替）

理想的には `message.created` 等のイベント配信とするが、**MVP では REST ポーリング**に置換する。将来、同じ認可モデルで WebSocket / SSE / マネージド Realtime に載せ替え可能とする。

---

## 9. 受入基準（MVP 完了の定義）

1. 2 つのブラウザセッションで**別ユーザー**としてログインし、**同一チャンネル**で双方向にメッセージが表示される。
2. ページリロード後、**過去メッセージが DB から復元**される。
3. **未認証**、または**非メンバー**が他ワークスペースのメッセージ API を呼ぶと **401 または 403** となる。
4. **不正なチャンネル名**および**重複名**はサーバーで拒否される。
